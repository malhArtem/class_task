
## Краткий словарь терминов

| Термин | Что означает |
|--------|---------------|
| **Репозиторий** | Папка, за которой Git следит. В ней хранится вся история изменений |
| **Коммит (commit)** | «Снимок» состояния всех файлов в определённый момент времени |
| **Ветка (branch)** | Параллельная линия разработки. Позволяет делать эксперименты, не трогая основную версию |
| **Удалённый репозиторий (remote)** | Версия проекта, которая хранится на сервере (например, на GitHub) |
| **Конфликт (merge conflict)** | Ситуация, когда Git не может автоматически объединить изменения из двух веток |



# Лабораторная работа: «Основы Git и GitHub»

**Цель работы:** Научиться сохранять историю изменений, создавать ветки для экспериментов, синхронизировать код с сервером (GitHub) и решать конфликты слияния.


---

## Часть 0. Подготовка: установка и настройка Git

Прежде чем начать работу, необходимо установить Git на ваш компьютер. Если вы работаете в Windows, скачайте установщик с официального сайта git-scm.com и запустите его. На macOS проще всего установить Git через Homebrew командой `brew install git`. На Linux используйте системный пакетный менеджер, например `sudo apt install git` для Ubuntu или Debian.

После установки нужно представиться системе Git. Это важно, потому что каждый ваш коммит будет подписан вашим именем и email. Выполните две команды:

```
git config --global user.name "Иван Петров"
git config --global user.email "ivan@example.com"
```

Ключ `--global` означает, что эти настройки будут использоваться для всех репозиториев на вашем компьютере. *Email должен совпадать с тем, который вы используете на GitHub*, чтобы коммиты правильно привязывались к вашему аккаунту.

Чтобы проверить, что настройки сохранились, введите `git config --list`. Вы увидите все параметры, включая имя и email.

## Часть 1. Создание локального репозитория

Теперь создадим папку, в которой Git будет хранить наш проект. На рабочем столе или в любой другой удобной папке создайте директорию с именем `recipe_book`. Откройте терминал (на Windows это может быть Git Bash, на macOS и Linux — обычный терминал) и перейдите в созданную папку командой `cd путь_к_папке/recipe_book`. Убедитесь, что вы находитесь внутри этой папки.

Теперь выполните команду `git init`. Она создаёт в текущей папке скрытую поддиректорию `.git`. Именно там Git будет хранить всю историю изменений, все ветки и служебную информацию. *Никогда не удаляйте эту папку вручную* — вы потеряете всю историю коммитов.

Проверить, что инициализация прошла успешно, можно командой `git branch`. Пока она ничего не выведет, потому что ещё нет ни одного коммита, но сам репозиторий уже готов к работе.

## Часть 2. Первый коммит

Создайте в папке `recipe_book` файл с именем `README.md`. Это обычный текстовый файл, но с расширением `.md`, что означает Markdown — лёгкий язык разметки. Откройте его в любом текстовом редакторе и напишите внутри:

```
# Книга рецептов
Проект для изучения Git
```

Символ `#` в начале строки означает заголовок первого уровня. Теперь сохраните файл.

Вернитесь в терминал и выполните `git status`. Эта команда показывает текущее состояние репозитория: какие файлы изменены, какие добавлены под контроль, а какие Git вообще не отслеживает. Вы увидите, что `README.md` отображается красным цветом в разделе «Untracked files» (неотслеживаемые файлы). Это значит, что Git знает о существовании файла, но ещё не начал следить за его версиями.

Чтобы начать отслеживать файл, нужно добавить его в так называемую «область подготовки» (staging area). Выполните `git add README.md`. После этого снова введите `git status`. Теперь файл станет зелёным и появится в разделе «Changes to be committed» (изменения, готовые к коммиту).

Теперь можно сохранить снимок состояния проекта. Это называется коммитом. Выполните `git commit -m "Initial commit: add README"`. Ключ `-m` позволяет указать сообщение коммита прямо в командной строке. Без него Git откроет текстовый редактор, где нужно будет вручную написать сообщение. *Старайтесь писать сообщения осмысленно* — они помогают понять, что было сделано в этом коммите.

Чтобы увидеть историю коммитов, введите `git log`. Вы увидите длинный хеш коммита (уникальный идентификатор), имя автора, дату и сообщение. Для более компактного вида используйте `git log --oneline` — тогда каждый коммит будет показан в одну строку.

## Часть 3. Добавление новых файлов и изменений

Теперь добавим первый рецепт. Создайте файл `borscht.md` и напишите в нём рецепт борща. Например:

```
# Борщ
Ингредиенты: свёкла, капуста, картошка, морковь, лук, мясо.
Приготовление: варить бульон, добавить овощи, варить 40 минут.
```

Сохраните файл и выполните `git status`. Вы снова увидите, что `borscht.md` — неотслеживаемый файл. Добавьте его в подготовку: `git add borscht.md`. Затем сделайте коммит: `git commit -m "Add borscht recipe"`.

Теперь представьте, что вы решили улучшить рецепт. Откройте `borscht.md` и измените время варки с 40 минут на 60 минут. Сохраните файл и снова выполните `git status`. На этот раз Git покажет, что файл `borscht.md` изменён (modified), но ещё не добавлен в подготовку. Это важное отличие: Git видит изменения в файлах, но не запоминает их автоматически. Вы должны сами решить, какие изменения войдут в следующий коммит.

Снова выполните `git add borscht.md`, а затем `git commit -m "Increase cooking time for borscht"`. Теперь у вас в истории три коммита (если считать самый первый с README). Проверьте это командой `git log --oneline`.

## Часть 4. Работа с ветками

Ветки — одна из самых мощных возможностей Git. Они позволяют вести параллельную разработку: вы можете создавать новую ветку для экспериментальной функции, и при этом основная ветка (обычно `main` или `master`) остаётся стабильной.

Посмотрите, какие ветки у вас есть прямо сейчас, командой `git branch`. Звёздочка `*` рядом с названием показывает, в какой ветке вы находитесь. Скорее всего, вы в `main` (или `master`, если Git настроен по-старому).

Создадим новую ветку для рецепта блинов: `git branch pancake`. Эта команда создаёт ветку, но не переключает на неё. Все ветки в Git — это просто указатели на определённые коммиты. Новая ветка `pancake` указывает на тот же самый коммит, что и `main`.

Теперь переключитесь на новую ветку: `git checkout pancake` (или более современная команда `git switch pancake`). Выполните `git branch` ещё раз — звёздочка теперь рядом с `pancake`.

Создайте файл `pancake.md` с рецептом блинов. Например:

```
# Блины
Ингредиенты: мука, молоко, яйца, сахар, соль, масло.
Приготовление: смешать все ингредиенты, жарить на сковороде.
```

Добавьте файл и сделайте коммит: `git add pancake.md`, затем `git commit -m "Add pancake recipe"`. Этот коммит существует только в ветке `pancake`.

Теперь переключитесь обратно на `main`: `git checkout main`. Посмотрите на содержимое папки — файла `pancake.md` нет! Это потому, что ветки изолированы друг от друга. Ветка `main` осталась в том состоянии, в котором была до создания `pancake`.

Объединим изменения из ветки `pancake` в `main`. Выполните `git merge pancake`. Git «перетянет» все коммиты из `pancake` в текущую ветку. После этого файл `pancake.md` появится и в `main`.

Теперь ветка `pancake` больше не нужна. Удалите её: `git branch -d pancake`. Ключ `-d` означает delete. Удаляется только указатель на ветку, сами коммиты остаются в истории. Проверьте `git branch` — осталась только `main`.

## Часть 5. Связь с GitHub (удалённый репозиторий)

Git позволяет синхронизировать локальный репозиторий с удалённым сервером. Самый популярный такой сервер — GitHub. Здесь мы будем хранить резервную копию нашего проекта и в будущем сможем работать совместно с другими.

Зайдите на сайт github.com и войдите в свой аккаунт (если у вас его нет, зарегистрируйтесь — это бесплатно). Нажмите зелёную кнопку «New repository» (новый репозиторий). В поле «Repository name» введите `recipe_book`. Остальные поля можете оставить по умолчанию, но *обязательно уберите галочку «Add a README file»* — мы уже создали README локально, и если GitHub создаст свой, возникнет конфликт. Нажмите «Create repository».

GitHub покажет страницу с инструкцией. Нас интересует раздел «…or push an existing repository from the command line» (отправить существующий репозиторий из командной строки). Там будут примерно такие команды.

Вернитесь в терминал, в папку `recipe_book`. Сначала нужно связать локальный репозиторий с удалённым. Выполните команду (заменив `ваш_логин` на ваш реальный логин на GitHub):

```
git remote add origin https://github.com/ваш_логин/recipe_book.git
```

Здесь `origin` — это стандартное имя для удалённого репозитория. Вы могли бы назвать его иначе, но `origin` используется по умолчанию. Чтобы проверить, что связь настроена, выполните `git remote -v`. Вы увидите две строки: fetch (загрузка) и push (отправка).

Теперь убедитесь, что ваша локальная ветка называется `main`. Если при инициализации репозитория у вас создалась ветка `master` (старое название по умолчанию), переименуйте её: `git branch -M main`. Ключ `-M` переименовывает ветку, а если ветки с новым именем нет, то создаёт.

Наконец, отправьте код на GitHub: `git push -u origin main`. Ключ `-u` (upstream) устанавливает связь между вашей локальной веткой `main` и удалённой веткой `main` на GitHub. После этого вы сможете использовать просто `git push` без дополнительных аргументов.

Обновите страницу репозитория на GitHub — вы увидите все ваши файлы: README.md, borscht.md, pancake.md. Репозиторий успешно синхронизирован.

## Часть 6. Внесение изменений через веб-интерфейс и получение их через pull

GitHub позволяет редактировать файлы прямо в браузере. Это полезно для мелких правок, когда нет доступа к компьютеру с Git. Заодно мы научимся забирать изменения с сервера.

На странице вашего репозитория на GitHub откройте файл `borscht.md`. Нажмите на иконку карандаша (Edit). Добавьте в конец файла новую строку: «Подавать со сметаной и зеленью». Прокрутите страницу вниз, там есть поле для сообщения коммита. Напишите «Add serving suggestion». Оставьте опцию «Commit directly to the main branch» (коммитить прямо в ветку main) и нажмите зелёную кнопку «Commit changes».

Теперь на GitHub файл изменился, а на вашем компьютере он всё ещё старый. Нужно получить эти изменения. В терминале выполните `git pull origin main`. Эта команда делает две вещи: сначала загружает изменения с удалённого сервера (`git fetch`), а затем объединяет их с вашей локальной веткой (`git merge`). После этого откройте локальный файл `borscht.md` — новая строка должна появиться.

## Часть 7. Конфликт и его разрешение

Конфликт возникает, когда две ветки изменяют одну и ту же строку в одном и том же файле, и Git не может автоматически решить, какой вариант правильный. Это нормальная ситуация при совместной работе, и важно уметь её разрешать.

Создадим конфликт искусственно. Сначала создайте новую ветку и сразу переключитесь на неё: `git checkout -b cheese_soup`. Ключ `-b` означает создание новой ветки и переход на неё.

Откройте файл `README.md`. Сейчас там, вероятно, всего две строки. Измените его, добавив новую строку: «Суп с сыром — в разработке». Сохраните файл, добавьте его в подготовку и сделайте коммит:

```
git add README.md
git commit -m "Add cheese soup note in README"
```

Теперь переключитесь обратно на `main`: `git checkout main`. В этой ветке измените ту же самую строку, но по-другому. Откройте `README.md` и замените ту же строку (ту, которую вы добавили в ветке `cheese_soup`) на «Супы: борщ, сырный суп». Сохраните, добавьте и закоммитьте:

```
git add README.md
git commit -m "Update soup list in main"
```

Теперь в двух ветках одна и та же строка выглядит по-разному. Попробуйте слить ветку `cheese_soup` в `main`: `git merge cheese_soup`. Git сообщит, что возник конфликт в файле `README.md`.

Откройте этот файл. Вы увидите что-то похожее на это:

```
<<<<<<< HEAD
Супы: борщ, сырный суп
=======
Суп с сыром — в разработке
>>>>>>> cheese_soup
```

Символы `<<<<<<< HEAD` показывают начало конфликтующего фрагмента. Часть между `HEAD` и `=======` — это то, что находится в текущей ветке (`main`). Часть между `=======` и `>>>>>>> cheese_soup` — это то, что пришло из ветки `cheese_soup`.

Ваша задача — *вручную исправить файл так, как должно быть правильно*. Удалите все маркеры конфликта (`<<<<<<<`, `=======`, `>>>>>>>`) и оставьте нужный текст. Например, вы можете объединить оба варианта, написав: «Супы: борщ, сырный суп (скоро в меню)». Или просто выбрать один из вариантов, если другой не нужен.

Сохраните файл. Теперь нужно сообщить Git, что конфликт разрешён. Добавьте исправленный файл в подготовку: `git add README.md`. Затем завершите слияние коммитом: `git commit -m "Resolve merge conflict in README"`.

Обратите внимание: *Git сам не создаёт коммит слияния в случае конфликта* — вы должны сделать это вручную. Теперь ветка `cheese_soup` успешно влита в `main`. Можете удалить её, если хотите: `git branch -d cheese_soup`.

Наконец, отправьте результат на GitHub: `git push`.

## Часть 8. Индивидуальное задание

Теперь каждый студент добавляет свой собственный рецепт в проект. Выберите один из вариантов или придумайте свой:

1. Пицца Маргарита
2. Греческий салат
3. Куриный суп с лапшой
4. Шоколадный торт
5. Овсяная каша с ягодами

*Порядок действий должен быть таким:*

Создайте ветку с именем `recipe_название_рецепта`, например `recipe_pizza`. В этой ветке создайте новый файл с именем рецепта (например, `margherita.md`) и напишите внутри ингредиенты и пошаговое приготовление. Добавьте файл и сделайте коммит с осмысленным сообщением. Затем отправьте ветку на GitHub: `git push -u origin recipe_pizza`.

Теперь зайдите на GitHub, в ваш репозиторий. Вы увидите сообщение о том, что только что запушенная ветка появилась, и ссылку на создание Pull Request. Нажмите «Compare & pull request». Напишите понятное описание того, что вы добавляете, и нажмите «Create pull request».

*Pull request* — это запрос на включение изменений из одной ветки в другую (в нашем случае — из вашей ветки в `main`). Преподаватель или другой студент может проверить ваш код, оставить комментарии и затем нажать «Merge pull request». После слияния ветка становится не нужна, и её можно удалить.

## Возможные ошибки и их решение

- *Ошибка «please tell me who you are»* при выполнении `git commit` — значит вы забыли настроить имя пользователя и email. Выполните `git config --global user.name` и `git config --global user.email`.

- *Ошибка «fatal: not a git repository»* — вы пытаетесь выполнить Git-команду в папке, которая не является репозиторием. Проверьте, что вы находитесь в правильной папке и что ранее выполняли `git init`.

- *«Merge conflict»* при слиянии — не паникуйте. Откройте проблемный файл, найдите маркеры `<<<<<<<`, исправьте содержимое вручную, затем выполните `git add` и `git commit`.

- *Ошибка «failed to push»* при попытке отправить код на GitHub — вероятно, удалённая ветка содержит изменения, которых нет у вас. Сначала выполните `git pull --rebase`, чтобы подтянуть изменения, а затем снова `git push`.

- *Если вы случайно начали писать сообщение коммита в редакторе и не знаете, как его сохранить* — в стандартном редакторе Vim (который часто открывается по умолчанию) нужно нажать `Esc`, затем ввести `:wq` и нажать Enter. Или можно сразу использовать `git commit -m "сообщение"`, чтобы избежать редактора.
