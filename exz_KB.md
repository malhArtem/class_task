## Экзаменационные билеты по разработке REST API на FastAPI

**Общие требования для всех билетов:**
- Язык реализации: Python 3.9+.
- Фреймворк: FastAPI.
- Данные хранятся в оперативной памяти (список словарей) или в БД.
- Автоматическая документация должна быть доступна по адресу `/docs`.
- Все эндпоинты должны возвращать корректные HTTP-статусы:
  - `200 OK` – успешный GET / PUT.
  - `201 Created` – успешное создание.
  - `204 No Content` – успешное удаление.
  - `404 Not Found` – ресурс не найден.
  - `400 Bad Request` – невалидные входные данные.
- При попытке получить, обновить или удалить несуществующую запись возвращать JSON с сообщением об ошибке, например `{"detail": "Entity not found"}`.
- Использовать Pydantic-модели для валидации запросов и ответов.
- Идентификатор `id` должен генерироваться автоматически (целое число, инкремент).

### Билет №1. Управление фильмами

**Описание:**  
Разработайте REST API для управления коллекцией фильмов. Каждый фильм содержит уникальный идентификатор, название и имя режиссёра.

**Модель данных:**
- `id` – целое число (генерируется автоматически).
- `title` – строка, обязательное поле, не может быть пустой.
- `director` – строка, обязательное поле, не может быть пустой.

**Эндпоинты:**
1. `GET /movies` – возвращает список всех фильмов в формате JSON.
2. `GET /movies/{movie_id}` – возвращает фильм по его `id`. Если фильм не найден – статус 404 и сообщение.
3. `POST /movies` – создаёт новый фильм. Принимает JSON с полями `title` и `director`. Возвращает созданный объект с присвоенным `id` и статус 201. Если поля не заполнены – статус 400.
4. `PUT /movies/{movie_id}` – полностью обновляет фильм. Принимает JSON с `title` и `director`. Если фильм существует – заменяет данные и возвращает обновлённый объект со статусом 200. Если не найден – статус 404.
5. `DELETE /movies/{movie_id}` – удаляет фильм. При успешном удалении – статус 204 без тела. Если фильм не найден – статус 404.

**Валидация:**  
Поля `title` и `director` должны быть непустыми строками (длина >= 1). При нарушении – статус 400.

### Билет №2. Управление пользователями

**Описание:**  
Разработайте REST API для управления пользователями. Каждый пользователь имеет идентификатор, имя пользователя и адрес электронной почты.

**Модель данных:**
- `id` – целое число (генерируется).
- `username` – строка, обязательное, непустое, уникальное (проверка на дубликаты при создании и обновлении – если имя уже существует, возвращать 400).
- `email` – строка, обязательное, должно содержать символ `@` (валидация формата email – можно использовать Pydantic `EmailStr`).

**Эндпоинты:**
1. `GET /users` – список всех пользователей.
2. `GET /users/{user_id}` – получить пользователя по id (404 при отсутствии).
3. `POST /users` – создать пользователя (принимает `username`, `email`). Возвращает созданного пользователя с id и статус 201. При дублировании username – 400.
4. `PUT /users/{user_id}` – обновить пользователя (принимает `username`, `email`). Заменяет полностью. Возвращает обновлённого. Если не найден – 404. При дублировании username – 400.
5. `DELETE /users/{user_id}` – удалить пользователя (204 или 404).

**Валидация:**  
`username` – непустая строка. `email` – валидный email (используйте `EmailStr` из Pydantic). При невалидном email – 400.

**Описание:**  
Создайте API для списка задач. Каждая задача имеет идентификатор, заголовок и необязательное описание.

**Модель данных:**

5. `DELETE /tasks/{task_id}` – удалить задачу (204 или 404).

**Валидация:**  
`title` – непустая строка (длина >= 1). `description` – может быть пустой строкой, но если передана, то должна быть строкой.
